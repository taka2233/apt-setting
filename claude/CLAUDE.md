# 不要な抽象化の禁止
関数抽出は「複雑性を隠して単純なインターフェースを提供する」ために行う。
呼び出し元にそのまま書いても開発者の認知負荷が変わらない関数は作成しないこと。
APoSDの原則に従ってください

## 禁止する関数の特徴
- 中身が1〜2行で、呼び出し先をそのまま転送しているだけ
- 引数の変換や条件分岐がなく、呼び出し元にインライン展開しても複雑性が増えない
- 関数名が呼び出し先のメソッド名と実質同じ意味を繰り返している

## 判断基準
「この関数を消してインライン化したとき、呼び出し元の可読性は下がるか？」を問う。
下がらないなら、その関数は間接参照のジャンプコストを増やしているだけなので不要。

### 例
```kotlin
// BAD: hostFilterSyncState.syncWith() を呼ぶだけのラッパー
private fun syncHostSelectionWithLatestData() {
    hostFilterSyncState.syncWith(availableHosts())
}
// GOOD: 呼び出し元に直接書く
val summaries = get()
allSummaries = summaries
hostFilterSyncState.syncWith(availableHosts())
methodFilterSyncState.syncWith(availableMethods())
```
抽出すべき場合
以下のいずれかに該当する場合は関数抽出が正当化される。

複数箇所から呼ばれ、ロジックの一元管理が必要
条件分岐やデータ変換を含み、呼び出し元の見通しが悪くなる
テスト対象として切り出す必要がある
将来の変更ポイントが明確で、変更時の影響範囲を局所化できる

もし私の指示が Shallow Module を生み出すようなものであった場合は、APoSDの観点からより良い（Deepな）代替案を提案してください。

# gitコマンドの制約
git add や git commit を実行する前に、必ずコミット内容を提示してユーザーの確認を取ること。確認なしにコミットを実行しないこと。

# Comment
役割：コードを読んでいるその瞬間の「迷い」をなくす。

コード自体で「何をしているか」がわかるのがベストですが、どうしても補足が必要な場合に書きます。

書くべきこと：
- 「なぜ（Why）」そうしたか： 複雑なロジックを選んだ理由、バグ回避のための特殊な処理。
- 意図（Intent）： コードから読み取りにくい、ビジネスルール上の背景。

注意点： 「ここを書き換えると〇〇に影響する」といった依存関係の警告。

書かないこと：
- 「何（What）」をしているか： i++; // iを1増やす のような、見ればわかること。
- 古いコード： コメントアウトして残さず、Gitに任せて削除しましょう。

# User確認の徹底
- Think in English and respond in Japanese.
- コードを修正する前にプランニングを行い、編集の方針をユーザに確認してからコードを編集してください。
- プランニングで作成するMarkdownは日本語で作成してください。
- 必要な情報があるときは公式ドキュメント等から取得してください。

# シェルコマンドについて
シェルコマンドは必要最小限にしてください。`echo "===== ... ====="` のような装飾、区切り、進捗表示、不要な `bash -lc`、長い `&&` 連結、複雑なワンライナーは避けてください。確認しやすいように、1コマンド1目的を原則とし、実行される処理が一目で分かるシンプルな形にしてください。
